# 技術調査 (ライブラリドキュメント取得)

ライブラリ、フレームワーク、SDK、API、CLI ツール、またはクラウドサービスについて質問された際は、`ctx7` CLI を使用して最新のドキュメントを取得する。

たとえ答えを知っていると思われる場合でも、学習データには最新の変更が反映されていない可能性があるため、この手順を優先する。ライブラリのドキュメントに関しては、Web 検索よりもこちらを優先して使用する。

## 適用する場合 / 適用しない場合

判定軸は「対象がライブラリかどうか」ではなく「質問の意図がライブラリ・フレームワーク・SDK・API・CLI の最新仕様や使い方を問うているか」。

- 適用する: API 構文・設定方法・バージョン移行・ライブラリ固有のデバッグ・セットアップ手順・CLI ツールの使用方法を問う質問。
- 適用しない: リファクタリング、ゼロからのスクリプト作成、ビジネスロジックのデバッグ、コードレビュー、一般的なプログラミング概念の質問。対象コードがライブラリを import していても、意図が上記なら起動しない。

## 実行手順

1. ライブラリの特定の為に以下を実行する。

   ```sh
   deno run --no-lock --allow-env --allow-sys --allow-read --allow-write --allow-net=context7.com,registry.npmjs.org npm:ctx7@latest library <name> "<user's question>"
   ```

2. 最適なライブラリ ID (形式: `/org/project`) を選択する。
3. ドキュメントを取得する為に以下を実行する。

   ```sh
   deno run --no-lock --allow-env --allow-sys --allow-read --allow-write --allow-net=context7.com,registry.npmjs.org npm:ctx7@latest docs <libraryId> "<user's question>"
   ```

4. 取得したドキュメントに基づいて回答する。

## 注意事項

- MUST: **クエリに機密情報 (API キー、パスワード、認証情報) を含めないこと**。
- ユーザから `/org/project` 形式で直接 ID が提供されない限り、必ず最初に library コマンドを実行して有効な ID を取得すること。
- 質問内容は、「nextjs」ではなく「Next.js」、「customerio」ではなく「Customer.io」、「threejs」ではなく「Three.js」など、正しい句読点を用いた公式のライブラリ名を使用すること。
- ライブラリの選択基準は次の優先順位で決定する。上位で差がつかない場合のみ下位で tie-break する。(1) 正式名称の一致 → (2) ソースの信頼性 (High / Medium を推奨) → (3) ベンチマークスコア (高いほど良い) → (4) 説明の関連性 → (5) コードスニペットの数。
- クエリにはユーザの質問全文を使用すること。具体的かつ詳細なクエリの方が、曖昧な単語よりも良い結果が得られる。
- 1 つの質問につき 3 回を超えてコマンドを実行しないこと。
- バージョン固有のドキュメントは、ユーザが特定バージョンを明示した場合、またはバージョン依存の挙動を問うている場合にのみ使う。その際は library の出力結果にある `/org/project/version` 形式 (例: `/vercel/next.js/v14.3.0`) を使用する。指定がなければバージョンなしの ID を使う。
- コマンドがクォータエラーで失敗した場合は、ユーザーにその旨を伝え、制限を引き上げるよう提案すること。勝手にトレーニングデータの内容で補完して回答を返すようなことはしないこと。
